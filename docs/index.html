Decode (M=1 GEMV) can't use the matrix cores — it's bandwidth-bound. Prefill (M=128 GEMM) is compute-bound, so coopmat is the lever there.

~73%

Decode reads ~6.5 GB of weights per token, so the ceiling is bandwidth ÷ 6.5. The realistic GPU ceiling is ~125 GB/s ÷ 6.5 ≈ 19 tok/s; the SoC's theoretical ~230 GB/s ÷ 6.5 ≈ 35 tok/s is a wall no GPU kernel reaches. The engine delivers 13.84 — ≈73% of the realistic ceiling (≈40% of theoretical). The gap is the strided-read penalty plus per-token fixed overhead (attention, norms, ~600 barriers), not the GEMV math.

The journey

From Google's own tooling failing to load, down through the NPU, to a from-scratch Vulkan engine — every step decided by measurement.

  1. Phase 1–5 · the off-the-shelf paths

    Google litert-lm won't load; the NPU is a dead end

    litert-lm hits a 256 MB buffer cap. The Hexagon NPU runs but produces the wrong token (fp16 overflow + QNN error 1003). The ONNX/QNN GPU path is correct but swap-bound at 0.024 tok/s. llama.cpp CPU (7.9) becomes the bar to beat.

  2. Phase 6–7 · WebGPU, then down to Vulkan

    A wgpu engine (7.07) hits the WebGPU wall

    996 compute passes rebuilt every token = ~100 ms of CPU encode. WebGPU can't record-once. Dropping to raw Vulkan and recording the entire ~600-dispatch token graph into one resubmitted command buffer eliminates it → ~10.7 tok/s.

  3. Phase 8 · the bandwidth pass

    Microbench-driven kernel rebuild → 13.84 tok/s

    Decode is a bandwidth-bound GEMV. Reinterpreting weights as 64-bit uvec2 loads (16 outputs/thread) lifts the load path 82→109 GB/s; parallel-softmax attention; adaptive K-split. +29%, to ~86% of the bandwidth roof.

  4. Phase 10–15 · context + prefill

    64k context and a 2.2× prefill

    Flash-decode attention + a sliding-window ring buffer reach 64k. Batched prefill on the coopmat matrix cores, then fp16 KV and an int8 (W8A8) GEMM — discovered because int8 coopmat is 3.7× the f16 path here — take prefill from ~95 to ~210 prompt-tok/s.

  5. Phase 16–21 · serving

    An OpenAI-compatible server an agent can drive

    Chat Completions + tool calling (Gemma's native format), a Responses API, a stateful session WebSocket, a transparent prefix KV cache (multi-turn drops to ~3–7 s/turn), and a degeneration guard. opencode runs against it.

The full log — every dead end, microbenchmark, and reversal — is in JOURNEY.md.

What it does

OpenAI-compatible API

/v1/chat/completions (streaming + tools), /v1/completions, /v1/responses, /v1/models.

Tool / function calling

Renders OpenAI tools to Gemma 4's native tool format and parses the calls back. Drives agent clients like opencode.

Stateful session WebSocket

/v1/sessions keeps the conversation server-side; the client sends only the next message. Rides the prefix cache.

Prefix KV caching

Auto-detects a shared prompt prefix, restores its KV with a ~ms GPU copy instead of re-prefilling — repeated prompts skip the prefix.

64k context

Flash-decode attention (online softmax, unbounded T) + a sliding-window ring buffer for the local layers. Fits in 48 GB.

Built-in profiler

Vulkan timestamp queries (PROFILE=1 / PREPROF=1) give per-kernel decode and prefill breakdowns.

Run it yourself

Native ARM64 Python 3.12 + the Vulkan SDK. One command sets up the env, downloads the weights, and builds the shaders.

# Windows ARM64
.\scripts\setup.ps1 -Model
.venv-gemma4\Scripts\python.exe src\serve.py
